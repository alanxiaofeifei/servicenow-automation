## 2. The exact UX problem

### What the user sees today

In the Release Readiness Handoff section of the workbench, the "Current package path" display reads:

```
\\wsl.localhost\Ubuntu-Compact\home\alanxwsl\projects\...
```

This path is generated by `formatPackagePathForDisplay()` at line 8560:

```ts
return `\\\\wsl.localhost\\Ubuntu-Compact${path.replace(/\//g, "\\")}`;
```

### Why this is a problem

1. **Bakes a personal WSL distro name into the renderer.** `Ubuntu-Compact` is Alan's WSL distro name. It will not work if:
   - Alan renames his WSL distro (`wsl --set-version Ubuntu-Compact ...` or similar)
   - The project is cloned on a machine with a different WSL distro
   - A new WSL installation uses `Ubuntu` (the default) instead of `Ubuntu-Compact`
   - Windows paths are accessed via a different mechanism (e.g., pure Windows, CI runner, or separate WSL instance)

2. **Already has a workaround that proves the fragility exists.** At line 4296, the clipboard-summary builder strips the hardcoded prefix with `replace(/^\\\\wsl\.localhost\\\\Ubuntu-Compact/, "")` — this is a clear sign that the codebase knows the prefix is machine-dependent but hasn't centralized the derivation.

3. **The adapters package already solves this correctly.** `browser-session.ts` in `packages/adapters/src/` has a private `resolveWslDistroName()` function (lines 1374–1378) that reads from `process.env.SDA_WSL_DISTRO ?? process.env.WSL_DISTRO_NAME` and validates the result. The adapters package generates correct UNC paths. The renderer does not reuse this logic.

4. **Two separate UNC path implementations.** The codebase now has at least two WSL→UNC conversion paths:
   - `packages/adapters/src/browser-session.ts` — dynamic, env-var-driven (correct)
   - `apps/desktop/src/App.tsx:formatPackagePathForDisplay()` — hardcoded (fragile)

### Non-breaking today

The hardcoded value matches Alan's WSL distro exactly, so the app works correctly on Alan's machine. This is not a crash, not a data leak, and not a regression. It is a **portability debt** that will surface the first time the app runs on a machine with a different WSL distro name.

## 3. Target state

### Before (current)

```
formatPackagePathForDisplay("/home/alan/.../package.zip")
→ "\\wsl.localhost\Ubuntu-Compact\home\alan\..."
                          ^^^^^^^^^^^^^^ hardcoded
```

### After (desired)

```
formatPackagePathForDisplay("/home/alan/.../package.zip")
→ "\\wsl.localhost\Ubuntu\home\alan\..."         // on an "Ubuntu" distro
→ "\\wsl.localhost\Ubuntu-Compact\home\alan\..." // on "Ubuntu-Compact"
→ "\\wsl.localhost\FedoraWSL\home\alan\..."     // on "FedoraWSL"
```

The UNC prefix is derived from runtime data (environment variables), not hardcoded. The derivation must:

1. **Check `SDA_WSL_DISTRO`** (project-specific override, set by `scripts/wsl/operator-env.sh`)
2. **Fall back to `WSL_DISTRO_NAME`** (standard WSL env var set by WSL itself)
3. **Validate** the value matches `[A-Za-z0-9_.-]+` (same validation as `browser-session.ts`)
4. **Provide a sensible fallback** if neither env var is available (e.g., not running inside WSL)

### What does not change

- The surrounding `formatPackagePathForDisplay()` signature: `(path: string | undefined, ok?: boolean): string`
- The fallback text for null/loading/unavailable states
- The test expectation that `\\wsl.localhost` appears in the output (only the distro segment changes)
- The clipboard-summary workaround at line 4296 (it becomes harmless once the prefix is derived, but removing it is a separate task)

## 9. Implementation notes for BD3

### BD3-A: Export `resolveWslDistroName` from adapters

Current state in `packages/adapters/src/browser-session.ts` (line 1374–1378):

```ts
function resolveWslDistroName(): string | undefined {
  const distroName = process.env.SDA_WSL_DISTRO ?? process.env.WSL_DISTRO_NAME;
  const sanitizedDistroName = distroName?.trim();
  return sanitizedDistroName && /^[A-Za-z0-9_.-]+$/.test(sanitizedDistroName)
    ? sanitizedDistroName
    : undefined;
}
```

Change: add `export` keyword. This function is private today; exporting it makes it available to renderer imports via `@servicenow-automation/adapters/browser`.

### BD3-B: Import and use in App.tsx

```ts
// Add import at top
import { resolveWslDistroName } from "@servicenow-automation/adapters/browser";

// Replace the hardcoded value in formatPackagePathForDisplay
// Line 8560 changed from:
//   return `\\\\wsl.localhost\\Ubuntu-Compact${path.replace(/\//g, "\\")}`;
// To:
//   const distro = resolveWslDistroName() ?? "WSL";
//   return `\\\\wsl.localhost\\${distro}${path.replace(/\//g, "\\")}`;
```

The fallback `"WSL"` is a safe generic placeholder when no WSL distro name is available (non-WSL environment). It produces `\\wsl.localhost\WSL\...` — visibly identifiable as a fallback rather than silently producing a wrong path.

### BD3-C: Remove clipboard workaround

Line 4296 currently has:

```ts
packageMetadata.path ? `path: ${formatPackagePathForDisplay(packageMetadata.path).replace(/^\\\\wsl\.localhost\\\\Ubuntu-Compact/, "")}` : null,
```

Once `formatPackagePathForDisplay` derives the prefix dynamically, the `replace()` is unnecessary. Change to:

```ts
packageMetadata.path ? `path: ${formatPackagePathForDisplay(packageMetadata.path)}` : null,
```

### BD3-D: Test updates

Line 1652 has:

```ts
expect(output).toContain("\\\\wsl.localhost");
```

This assertion still passes because the output will still contain `\\wsl.localhost`. However, if the test environment does not set `SDA_WSL_DISTRO` or `WSL_DISTRO_NAME`, the distro segment will be `WSL` (fallback) instead of `Ubuntu-Compact`. The test should be updated to either:

- Set `process.env.SDA_WSL_DISTRO = "Ubuntu-Compact"` before the test and restore after, OR
- Use a regex assertion: `expect(output).toMatch(/\\\\wsl\.localhost\\.+/);`
